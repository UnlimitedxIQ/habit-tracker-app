# Architecture — Habit Tracker

## System Design

```
┌───────────────────────────────────────────────────────┐
│                   Vercel Platform                       │
│            (Framework Preset: Services)                 │
│                                                         │
│  ┌──────────────────┐      ┌─────────────────────────┐ │
│  │   Next.js 16     │      │   FastAPI (Python)       │ │
│  │   apps/web/      │─────▶│   backend/main.py        │ │
│  │   routePrefix /  │      │   routePrefix /server    │ │
│  │                  │      │                           │ │
│  │  Dashboard SPA   │      │  /health                  │ │
│  │  shadcn/ui       │      │  /habits CRUD             │ │
│  │  SWR hooks       │      │  /habits/{id}/toggle      │ │
│  │  Tailwind CSS    │      │  /completions/weekly      │ │
│  └──────────────────┘      └───────────┬───────────────┘ │
│                                         │                 │
│                                  ┌──────┴──────┐          │
│                                  │   SQLite    │          │
│                                  │   habits.db │          │
│                                  └─────────────┘          │
└───────────────────────────────────────────────────────────┘
```

## Tech Stack

| Layer | Choice | Rationale |
|-------|--------|-----------|
| Frontend | Next.js 16 (App Router) | Server Components, file-system routing |
| UI Kit | shadcn/ui + Tailwind CSS 4 | Design-system components, CSS variables |
| Fonts | Sora + DM Sans + JetBrains Mono | Distinctive, warm personality |
| Data Fetching | SWR (`useSWR`) | Client-side with optimistic updates, revalidation |
| Backend | FastAPI (Python 3.12) | Async, auto OpenAPI docs, fast |
| Database | SQLite via `aiosqlite` | Zero config, single file, MVP-appropriate |
| ORM | None — raw SQL | 2 tables, ~10 queries, simplicity wins |
| Deployment | Vercel Services | Single project, shared routing, auto env vars |

## Vercel Services Configuration

```json
{
  "experimentalServices": {
    "web": {
      "entrypoint": "apps/web",
      "routePrefix": "/"
    },
    "api": {
      "entrypoint": "backend/main.py",
      "routePrefix": "/server"
    }
  }
}
```

## API Communication

- Frontend calls backend via `NEXT_PUBLIC_API_URL` (auto-generated by Vercel Services as `/server`)
- In local dev, `vercel dev -L` runs both services and handles routing automatically
- Backend routes are defined WITHOUT the `/server` prefix — Vercel strips it before forwarding
- For local dev without Vercel CLI: frontend uses `http://localhost:8000` fallback
- No CORS needed in production (same domain); CORS enabled in dev for localhost cross-port

## Key Decisions

1. **Vercel Services over separate deploys**: Single project, shared domain, auto-provisioned URL env vars. No CORS in prod.
2. **SQLite over Postgres**: MVP doesn't need multi-user. Zero config, fast for single-user.
3. **Raw SQL over ORM**: Only 2 tables, ~10 queries. ORM adds complexity without value here.
4. **SWR over Server Components for data**: Habit toggles need instant optimistic updates. SWR handles mutation + revalidation.
5. **Hard delete over soft delete**: MVP simplicity. CASCADE deletes completions with the habit.
6. **No auth**: Single-user MVP — no sensitive data.
7. **shadcn/ui over custom components**: Production-quality primitives (Sheet, Dialog, AlertDialog, DropdownMenu) save time.
